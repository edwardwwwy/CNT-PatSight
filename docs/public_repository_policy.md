# GitHub 公开发布规则

本规则的目标是让 GitHub 仓库保持轻量、可复现和可审计，同时避免公开受限论文、完整数据库、内部研发数据与凭据。公开仓库采用白名单思路：没有明确发布理由和权利依据的数据，默认保留在本地。

## 推荐公开结构

```text
CNT-PatSight/
├── README.md
├── LICENSE                       # 由项目所有者选择后加入
├── requirements.txt
├── config/                       # 不含真实密钥
├── data/
│   ├── samples/                  # 1–3 个审核后样例
│   ├── processed/templates/      # 空白模板
│   └── review/screening_benchmark/
├── docs/
├── reports/                      # 仅公开、汇总后的报告和图表
├── scripts/
└── tests/                        # 小型公开 fixture
```

## 公开白名单

- 采集、解析、提取、校验与报告代码；
- schema、字段字典、筛选规则、无密钥配置和 `.env.example`；
- 单元测试以及合成、脱敏或明确允许公开的 fixture；
- benchmark 的抽样设计、汇总指标、错误类型和经检查的审计表；
- 1–3 个通过人工复核与权利检查的八表样例；
- DOI、题名、作者、开放获取链接、获取日期和许可证元数据；
- 不包含受限内容的流程图、统计图和项目报告。

## 默认不公开

- `.env`、API key、访问令牌、密码、私钥、服务账户文件；
- 公司实验数据、内部配方、未公开研发记录和带身份信息的日志；
- 付费、学校权限或非授权来源获得的论文 PDF；
- 未确认再分发权利的开放论文副本和补充材料；
- 原始 API 响应、全文 HTML/文本、OCR 输出和大段原文摘录；
- 完整元数据库、SQLite、队列、租约、缓存、模型权重和批量中间结果；
- 本地绝对路径、调试转储、交付目录和未脱敏人工复核队列。

开放获取只说明访问条件，不自动授予重新分发文件的权利。确需复制采用 CC BY 等开放许可证的文件时，也要确认版本与条款，并保留标题、作者、来源和许可证归属。默认方案仍是保存 DOI、OA URL 与许可证元数据，让使用者从合法来源自行获取全文。

## 大型数据库发布

完整数据库达到稳定版本后，应单独冻结并记录：

- 数据版本、生成日期、schema 版本和代码提交；
- 来源范围、纳入标准、审核覆盖率和已知局限；
- 数据许可证及与第三方来源权利的关系；
- 文件哈希、行数、格式和复现命令。

大型 CSV/Parquet 优先发布到 GitHub Release、Zenodo 或数据集平台，不进入普通 Git 历史。GitHub 会阻止超过 100 MiB 的普通 Git 单文件；更小但频繁变化的数据库和生成数据同样会快速膨胀仓库历史。参见 [GitHub 大文件说明](https://docs.github.com/en/repositories/working-with-files/managing-large-files/about-large-files-on-github)。

## 发布前检查

1. 运行测试和八表校验，确认公开样例可以复现。
2. 用 `git status --short` 和 `git diff --cached --stat` 检查即将提交的范围。
3. 用 `git ls-files` 检查是否仍跟踪 PDF、数据库、完整中间数据和交付目录。
4. 搜索密钥、令牌、邮箱密码、内部域名、用户名和本地绝对路径；只保留空值示例。
5. 检查 CSV、JSON、Excel 和日志中的隐藏 sheet、批注、路径、作者元数据和原文摘录。
6. 为每个公开样例记录来源、许可证、审核人/审核状态和生成版本。
7. 确认 README 中的命令、相对链接和 benchmark 数字与当前版本一致。
8. 最后检查暂存区，而不只检查工作目录或 `.gitignore`。

## 已被 Git 跟踪的数据

`.gitignore` 对已经跟踪的文件无效。若现有仓库准备首次公开，应在保存当前工作后，用 `git rm --cached` 将 `data/raw/`、`data/interim/`、`data/derived/`、`data/review/extraction/` 和 `output/` 中的非公开内容移出 Git 索引，再单独加入白名单目录。`--cached` 会保留本地文件，但会改变暂存区，因此必须先确认没有需要保留的暂存改动，并逐项检查 `git status`。

如果非公开内容只存在于**尚未推送**的本地提交中，应先重写该本地提交，再 push；仅在后续提交中删除文件，仍会把含数据的早期提交上传到远端。重写前应保存工作目录和暂存区状态，并确认基准分支与目标文件清单。不要在存在未确认改动时直接执行 reset、rebase 或强制删除索引。

如果受限文件已经推送到远端，仅在新提交中删除并不能从历史中移除它。此时应先停止继续传播、轮换可能泄露的密钥，再由仓库所有者协调历史重写和协作者重新同步。GitHub 的相关说明见 [从仓库中移除敏感数据](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository)。

## 许可状态

代码许可证与数据许可证应分开选择，因为第三方文献权利不会随项目代码许可证一并转授。当前仓库没有 `LICENSE` 文件；在项目所有者明确选择并加入许可证前，不应对外宣称项目已经开源。
